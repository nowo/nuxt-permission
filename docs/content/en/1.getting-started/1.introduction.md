**nuxt-permission** brings backend menu / permission driven dynamic routes to Nuxt: non-whitelisted pages are stripped from the static route table at build time and registered from the backend menu **after login** — before that the routes simply do not exist. It ships permission state, `hasPermission`, and a menu-transform helper, with types extensible via declaration merging.

## How it works

1. At build time, every page **not** in the `static` whitelist is removed from the route table and collected into a code-split `import()` manifest.
2. On first paint (refresh / deep-link), the library runs your data source inside `router.options` — SSR-safe, before the first navigation.
3. After login you call `load()`, which runs the same source, writes permission state, and registers the routes from the backend menu.

## What the library handles

- Stripping non-whitelisted routes at build time and registering them at runtime
- Permission / menu state (`usePermissionState`)
- The menu-transform helper (`normalizeMenus`)
- `hasPermission` for UX-level permission checks

## What is up to you

- Token storage, login / logout endpoints
- The redirect for unauthenticated users (the library only guarantees that non-whitelisted routes cannot be reached)

::callout{icon="i-lucide-shield-alert" color="warning"}
`hasPermission` is a **UX layer, not a security boundary** — real authorization always lives on the backend.
::

`normalizePermissionMenus` turns the raw backend menu tree into route shape:

- a node marked `_btn: true` in `cb` is **folded** into the parent's `meta._permission` (key = the `permission` field, value = the whole button node with the `_btn` marker stripped)
- a node with menu children is treated as a **group** and `redirect`s to its first child
- `cb` returning a falsy value **drops** the node and its subtree
- a backend `meta` object is **flattened** into `meta`

```ts
const tree = normalizePermissionMenus(menus, v => ({ ...v, _btn: `${v.type}` === '2' }))
```

The second argument `cb` is optional; omit it when the backend data is already in shape and there are no button permissions to fold.

::callout{icon="i-lucide-lightbulb"}
`cb` runs on **every node** â€” it is the single place to reshape **any** field: rename, add, drop, or transform. Marking `_btn`, remapping `name` â†’ `title` (below), normalizing a numeric `type` â€” they are all just field tweaks inside `cb`.
::

## Reserved keys

All library keys are `_`-prefixed and isolated from your backend data:

| Key | Where | Meaning |
| :-- | :-- | :-- |
| `_btn` | input (you set it in `cb`) | marks a node as a button to fold |
| `meta._permission` | output | folded button permissions on the parent |
| `meta._external` | output | flags an external link (rendered as `<a>`, not registered) |

## Menu path handling

- **External links** (`http(s)://`, protocol-relative `//`) are kept in the tree but not registered â€” render them as `<a>` (they carry `meta._external`).
- **Param pages**: a backend path may use bracket or colon syntax â€” `/detail/[id]` and `/detail/:id` both match `pages/detail/[id].vue`.
- **Query strings**: `/report?range=7d` registers the pathname `/report` while keeping the full value for the sidebar link.
- **Deduplication**: the same page reached from several menu entries registers once (first wins); every entry still shows in the menu tree.

## Route-level fields & the `name` caveat

Fields that are valid vue-router route keys â€” `path`, `name`, `redirect`, `alias`, `props`, `beforeEnter`, `sensitive`, `strict` (plus any you add via the `routeFields` option) â€” are kept at the **route level**; everything else on the node goes into `meta`.

::callout{icon="i-lucide-triangle-alert" color="warning"}
`name` is one of them, so a backend `name` **overrides the page's own route name** (it resolves to `node.name ?? entry.name` â€” backend wins). vue-router route names must be **unique**, but entries here are de-duplicated by `path`, not `name`. If two menu entries share a `name` (e.g. the same display label), vue-router warns about a duplicate name and name-based navigation (`navigateTo({ name })`) becomes ambiguous.
::

If your backend `name` is a **display label** (not a unique id), remap it in `cb` so it does not become the route name:

```ts
normalizePermissionMenus(menus, v => ({
    ...v,
    title: v.name, // display text â†’ meta.title (not a route-level field)
    name: undefined, // don't override the route name â†’ the page's own name is used
    _btn: `${v.type}` === '2',
}))
```

Then render the sidebar with `meta.title`. Alternatively, have the backend send a unique `name` (e.g. `sys-user`) plus a separate `title` for display.

## Migrating from 1.x to 2.0

::callout{icon="i-lucide-triangle-alert" color="warning"}
**Breaking changes in 2.0:**
- `normalizeMenus` â†’ **`normalizePermissionMenus`** (this function was renamed)
- `setMenusList` â†’ **`setMenuList`** (on `usePermissionState()` and the source context)
- buttons are marked with **`_btn`** in `cb`, no longer inferred from `type === 'button'`
- folded permissions moved from `meta.permission` â†’ **`meta._permission`**

So: rename the function/setter calls, set `_btn` in your `cb`, and read `route.meta._permission`.
::
